REST API Design

Design af et REST API det handler om definition af endpoints navngivning, rækkefølge af HTTP metoder, altså hvilke HTTP metoder der skal være først, i midten og til sidst, og hvad de skal bruges til fx. GET liste af alle books. Her er beskrivelse og et eksempelt, på REST API Design på drinks:
1. use the appropriate http method for the action
2. the ordering of the http methods
3. naming the endpoints so that they map to the collection in your system
HTTP methods Endpoints Usage
GET /drinks get all drinks
GET /drinks/:id get a drink by :id
POST /drinks create a new drink
PUT /drinks/:id update the entire drink resource
PATCH /drinks/:id update parts of the drink resource
DELETE /drinks/:id delete a drink resource by id
Det er de regler man skal følge, når man laver et REST API Design, både at bruge de rigtige HTTP-metoder (GET, POST, PUT, PATCH, DELETE), til at create, read, update og delete data (CRUD).

Rækkefølgen på http metoder er altid som du kan se ovenfor i tabellen, hvor endpoints er altid navnetord / substantiv i fleretal fx. drinks. Rækkefølgen organiseres altid på en logisk og konsistent måde for at gøre det lette at navigerer rundt og forståelse mæssige af hvor det er i koden og hvordan det er programmeret.

Det kan også være at det skal bruges sendes til udvikling eller til at vedligeholde i længere perioder ude i fremtiden.
Navngivningskonventioner er altid i fleretal til de forskellige endpoints, som gøre det nemmere at forstå og bruge, i stedet for man skal huske hvor det er ental eller fleretal eller om det er et helt andet ord. På den måde kan man altid vide at det afspejler den underliggende datamodel fx. drinks og at endpoints ikke kan være andre ord hvis det er drinks api.
